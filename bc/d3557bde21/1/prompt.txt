### H-01: MCP API Key Hashing Uses Single-Pass SHA-256 Without Proper KDF

**File:** `/Users/flake/sources/misc/secman/src/backendng/src/main/kotlin/com/secman/service/McpAuthenticationService.kt` (lines 227-236)

**Description:** API key secrets are hashed using a single pass of SHA-256 with the keyId as salt. This is insufficient for secret storage because SHA-256 is fast and GPU-accelerable. An attacker with database access can brute-force API key secrets efficiently.

```kotlin
fun hashApiKeySecret(secret: String, keyId: String): String {
    val salt = keyId.toByteArray(Charsets.UTF_8)
    val secretBytes = secret.toByteArray(Charsets.UTF_8)
    val digest = MessageDigest.getInstance("SHA-256")
    digest.update(salt)
    digest.update(secretBytes)
    return digest.digest().joinToString("") { "%02x".format(it) }
}
```

**Impact:** If the database is compromised, API key secrets can be recovered via brute force or rainbow tables.

**CWE Reference:** CWE-916 (Use of Password Hash With Insufficient Computational Effort)

**Recommended Fix:**
Use BCrypt, Argon2, or PBKDF2 with a high iteration count. Since API keys have high entropy (48 characters from a 64-character alphabet), a faster KDF like HMAC-SHA256 with a server-side secret key (keyed hash) is also acceptable, but a proper KDF is preferred.

### H-02: MCP API Key Authentication Iterates All Active Keys (Timing Side Channel)

**File:** `/Users/flake/sources/misc/secman/src/backendng/src/main/kotlin/com/secman/service/McpAuthenticationService.kt` (lines 80-111)

**Description:** The `authenticateApiKeyBySecret` method loads all active API keys from the database and iterates through them, comparing SHA-256 hashes. This creates two issues:
1. **Timing attack**: The time to authenticate varies with the number of keys and position of the matching key, potentially leaking information.
2. **Performance/DoS**: Loading all active keys into memory for every authentication attempt is inefficient and could be exploited for resource exhaustion.

**Recommended Fix:**
Store a non-sensitive prefix (first N bytes) of the key alongside the hash for efficient lookup, then verify the full hash. Alternatively, use a keyed-hash approach where the key ID is derivable from the secret.
