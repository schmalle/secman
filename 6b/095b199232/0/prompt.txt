# Fix: CLI Startup Triggers Backend Migration Services

## Context

Running `./bin/secman manage-user-mappings list-bucket` boots the **full Micronaut application context** ‚Äî including Flyway, Hibernate, HikariCP, and 4 backend startup listeners ‚Äî even though the command only needs `S3DownloadService`. The `RequirementIdMigrationService` fires @Async on a background thread; by the time it executes, the `.use { }` block has already closed the context, causing a "Session/EntityManager is closed" stack trace after the S3 listing completes successfully.

**Root cause**: `SecmanCli.kt` line 182 calls `ApplicationContext.run()` which triggers full bean scanning of the `backendng` module. No mechanism distinguishes CLI mode from backend mode.

**Affected startup listeners** (all in backendng, all do DB operations on every CLI invocation):
1. `RequirementIdMigrationService` ‚Äî @Async @EventListener(StartupEvent), queries+updates all requirements (**crashes**)
2. `DefaultAdminBootstrapper` ‚Äî creates default admin user if none exist
3. `SystemUseCaseInitializer` ‚Äî creates IT/OT/NT use cases
4. `ProviderInitializationService` ‚Äî creates GitHub/Microsoft identity providers with placeholder credentials
5. `FlywayRepairCustomizer` ‚Äî connects to DB to inspect/repair `flyway_schema_history`

None of these are needed by any CLI command.

## Recommended Approach: Micronaut "cli" Environment

Use `ApplicationContext.builder().environments("cli").start()` instead of `ApplicationContext.run()`. Then suppress backend-only startup beans with `@Requires(notEnv = ["cli"])` and disable Flyway via environment-specific YAML.

This is the standard Micronaut mechanism for exactly this use case. It fixes the crash, eliminates unnecessary startup work for ALL CLI commands, and is easily maintained (future startup listeners just add the same annotation).

## Implementation Steps

### 1. SecmanCli.kt ‚Äî Use "cli" environment for all context creations

Extract a helper to avoid repetition across 9 call sites:

```kotlin
private fun createCliContext(): ApplicationContext {
    return ApplicationContext.builder().environments("cli").start()
}
```

Replace all `ApplicationContext.run().use { ctx ->` with `createCliContext().use { ctx ->`.

Lines to change: 182, 191, 200, 209, 218, 227, 236, 245, 253.

### 2. Create `application-cli.yml`

New file at `src/cli/src/main/resources/application-cli.yml`:

```yaml
flyway:
  datasources:
    default:
      enabled: false
```

Micronaut automatically loads `application-{env}.yml` when an environment is active. This disables Flyway migrations when running as CLI (the database is expected to be already migrated by the backend).

### 3‚Äì7. Add `@Requires(notEnv = ["cli"])` to 5 backend beans

Each file gets one import + one annotation before `@Singleton`:

```kotlin
import io.micronaut.context.annotation.Requires

@Requires(notEnv = ["cli"])
@Singleton
```

Files:
- `src/backendng/src/main/kotlin/com/secman/service/RequirementIdMigrationService.kt` (line 23)
- `src/backendng/src/main/kotlin/com/secman/config/DefaultAdminBootstrapper.kt` (line 23)
- `src/backendng/src/main/kotlin/com/secman/config/SystemUseCaseInitializer.kt` (line 11)

## Why Not the Alternatives

| Approach | Why not |
|----------|---------|
| **Bypass context for S3-only commands** | Only fixes `list-bucket`; `import-s3` still needs DB. Creates divergent code paths. Doesn't help other CLI commands. |
| **Disable beans via custom properties** | Non-standard; Micronaut environments exist for this. Harder to discover/maintain. |
| **Fix only RequirementIdMigrationService** | Leaves 3 other listeners doing unnecessary DB work every CLI run. `ProviderInitializationService` actually creates identity providers with placeholder credentials ‚Äî actively harmful. |

## Risk Assessment

- **No CLI command depends on these startup listeners** ‚Äî verified by grep. CLI commands use repositories directly, not via startup-initialized state.
- **The "cli" environment is additive** ‚Äî all beans without `@Requires(notEnv)` continue loading normally. Database, JPA, and repositories remain available for commands that need them.
- **Backend is unaffected** ‚Äî the backend does not use the "cli" environment, so all 5 beans continue to load normally there.
